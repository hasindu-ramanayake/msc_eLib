# NotificationService

## Description
The `NotificationService` handles user-centric communications within the `msc_eLib` platform. It collects events generated by other microservices (such as account creation, borrow confirmations, and overdue alerts) and stores them as notifications for the user. It provides an API for users to retrieve and manage their notification history.

## Tech Stack
- **Framework**: Spring Boot 3.x
- **Language**: Java 21
- **Database**: H2 (In-memory for lightweight persistence)
- **Messaging**: RabbitMQ (Consumes events from all business services)

## API Endpoints

| Method | Endpoint | Description |
| :--- | :--- | :--- |
| `GET` | `/api/v1/notifications/users/{userId}` | List all notifications for a user (newest first) |
| `PATCH` | `/api/v1/notifications/users/{userId}/read-all` | Mark all unread notifications as read for a user |
| `POST` | `/api/v1/notifications/events` | Internal endpoint to manually trigger a notification event |

## Key Components
- **NotificationController**: Exposes endpoints for users to view and manage notifications.
- **EventConsumer**: The RabbitMQ listener that captures diverse system events (User Created, Item Borrowed, etc.) and transforms them into user notifications.
- **NotificationRepository**: Persistence layer (JPA) for storing notification messages.

## Dependencies
- **DiscoveryServer**: Registered for service-to-service communication.
- **RabbitMQ**: Essential for receiving asynchronous triggers from other services.
- **UserService**: Provides context for user IDs and notification targeting.
- **ApiGateway**: Proxies requests from the web frontend to the notification API.
